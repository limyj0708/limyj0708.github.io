제목에 (1)이 붙은 이유는, 수집한 모든 데이터를 사용하여 모델을 학습시킨 것이 아니기 때문.
Hyperparameter 튜닝도 진행하지 않았음.
(2)에서는 모든 데이터를 사용하여 학습하고, 결과가 가장 좋게 나온 모델에 Hyperparameter 튜닝도 진행할 예정.

1. 학습에 사용한 데이터

  • 상세 수집조건은 <기업부도예측-1> 프로젝트 개요, 수집 데이터 참조
  • 2606개 기업의 3년치 재무 데이터
    • 생존 기업은 2019/12/31 기준 최신 3년치, 상장폐지된 기업은 상장폐지일 기준 최신 3년치
    • 사용한 feature
      • 부채비율
      • 자기자본비율
      • 현금자산비율
      • 당기순이익율
      • 총자산회전율
      • 총자산증가율(전년동기)
      • 당기순이익증가율(전년동기)
  • 2606개 기업의 지난 3개 감사보고서 데이터
    • 사용한 feature
      • 지난 3개 감사보고서에서, ‘계속기업 불확실성’관련 키워드가 언급된 경우 1, 아니면 0으로 표시
  • 상장시장 (코스닥, 코스피)

1-1. Oversampling

  • 전체 2606개 기업 중, 부도 기업은 전체의 16.04%
    • 불균형한 데이터(Imbalanced data)
    • Imbalanced data를 그대로 학습시키면, Recall이 낮게 나옴. Precision이 좀 낮아도 모든 부도 기업을 잡아내는 것이 좋은 부도 예측이라는 주제에서는, 허용할 수 없는 일
  • Imbalanced-learn 패키지를 사용하여, 소수 클래스를 증폭시키는 Oversampling을 진행
    • RandomOverSampling 사용 : 소수 클래스의 데이터를 반복해서 넣는 방법
    • 가상 데이터를 생성하는 Adaptive Synthetic Sampling, SMOTE 기법을 사용하지 못한 이유
      • 재무데이터에서 ‘수치를 계산할 수 없는 경우’에는, 구분을 위해 동종의 데이터가 절대로 절대로 가질 수 없는 큰 값을 할당하였음(100,000% 등)
        • 예 > [ 부채비율 = (유동부채 + 비유동부채)/자기자본 ] 에서,완전자본잠식이 되어 자기자본이 0이 된 경우
      • 가상의 값을 생성하게 되면, 등장해서는 안 되는 50,000%, 70,000% 등의 값이 생성됨

2. 모델 학습

  • Decision Tree 기반의 다음 세 모델을 사용하였음
    • RandomForest
    • GradientBoosting
    • XGBoost
  • 각 모델들의 상세 설명은 다른 게시글에서 정리

2-1. StratifiedSuffleSplit

  • 정말 기가 막힌 운으로 데이터가 분할되어 성능이 좋게 나온 모델이 있었기 때문에, StratifiedSuffleSplit으로 Train set과 Test set 100건에 대한 검증 진행
  • 앞으로도 한참 써야 하기 때문에 별도의 모듈을 만들어서 진행
    • Oversampling, 학습, 예측, 평가결과 출력을 한 번에 진행함
    • 모듈 코드는 Github에서 확인 가능

2-2. 결과비교

  • 학습 코드는 Github에서 확인 가능
  • 요약 : 하나의 모델을 사용해야 한다면, 가장 Recall이 높은 GradientBoost-RandomOversampling 데이터 학습 모델을 선정

2-2-1. Oversampling 진행하지 않은 Raw 데이터

<기업부도예측-2> GCP compute instances + shell script로 경제기사 크롤링

기업부도예측 프로젝트를 진행하기 위해 고른 데이터 중 하나는 경제지 기사 데이터다.
한국ABC협회 기준, 2018 일간 경제지 유료부수 판매 Top5에 드는 신문사에서 2708개 기업의 3년치 기사를 크롤링했다. 수집한 총 기사 수는 약 400만개로, 이전에 비동기로 40만개를 크롤링 했을 때보다 10배의 기사를 수집하게 되었는데, 이 기사들을 수집하는 데 GCP compute instance를 어떻게 활용하였는지 정리하였다.

비동기 크롤링 내용은 이미 충분히 다루었으니, gcloud SDK 명령어와 Shell script에 대한 내용이 되겠다.

1. 어떻게 활용하였는가?

  1. 발행일/제목/본문을 수집해야하는 기사들의 URL 준비 (이하 URL.csv)
  2. 기사 URL을 패러미터에 전달하면, 발행일/제목/본문을 수집하는 코드 작성 (이하 body_crawl.py)
  3. 32개의 Compute instance 생성 (Ubuntu 18.04) 및 환경세팅
  4. URL.csv를 64개로 분할하여 각 instance에 업로드
  5. body_crawl.py를 각 instance에 업로드
  6. 수집 시작 : 결과물은 csv 파일
  7. 수집 결과 파일들을 로컬 머신에 다운로드
  8. 수집 결과 파일 병합
  9. 끝
    비고 > 32개를 한 번에 실행 시, 몇몇 instance가 불안정한 문제가 있어서 최종적으로는 8개를 사용했다.

2. 한 걸음씩 살펴보기

2-1. Google cloud SDK 설치

  • Shell에서 다량의 instance를 제어하기 위해 SDK 설치
  • https://cloud.google.com/sdk/docs/quickstart-macos

2-2. URL.csv 분할

  • 아래 shell script를 분할하려는 csv 파일과 같은 폴더에 넣고 실행하자.
  • split 명령어에 대한 더 상세한 설명은 여기를 참고하자.

2-3. Compute instance 생성

  • GCP Console에서 Compute instance를 하나 생성하여 환경세팅까지 완료하자. 이 instance로 이미지를 생성하여, 이미지를 기반으로 나머지 instance를 한 번에 만들 것이다.
  • gcloud compute images create "cw-machine" --source-disk "cw-machine-1"
    • cw-machine-1 instance를 기반으로, cw-machine 이미지를 만든다.
  • gcloud compute instances create "cw-machine-2" --image-project "ml-practice-261005" --image "cw-machine"
    • cw-machine 이미지에서 cw-machine-2 instance를 만든다.
    • 이 명령어를 기반으로, 여러 instance를 한 번에 실행하기 위핸 shell script를 만들었다.

2-4. 크롤링 코드, 분할한 URL 파일 업로드

2-4-1. 크롤링 코드 업로드

  • 파일 하나를 업로드 하는 건 별 일 아니다.
  • gcloud SDK 명령어만 좀 바뀌었을 뿐, instance 생성 시의 코드와 별로 다르지 않다.

2-4-2. 분할한 URL 파일 업로드

  • 머신 이름을 받아와서 URL 파일과 크롤링 코드에 사용했으면 그냥 올려도 되겠으나, 크롤링 코드가 URL.csv를 읽게 되어있기 때문에 각 머신에 분할된 URL 파일(URL_1.csv…)을 올릴 때 이름을 URL.csv로 바꿔주어야 한다.
  • 그 외에는 유사함

2-5. 수집 실행

  • 모든 머신에서 한 번에 명령을 실행시킬 때 두 가지 방법을 고려했다.
    • iterms2 같은 multi-pane, multi-tab 지원 ssh 클라이언트에서 모든 머신에 한 번에 접속하여 동시에 명령 보내기
    • gcloud SDK로 머신에 명령어 보내기
  • ssh 클라이언트에서 보내는 것은, 모든 머신에 로그인하는 스크립트를 짜기 번거롭기도 하고 화면에 뭐가 계속 떠 있으니, gcloud SDK로 명령어를 보내 보자.
  • 크롤링 완료 알림은, 크롤링 코드 내에서 텔레그램 봇으로 메세지를 보내는 것으로 처리했다.

2-6. 수집 결과 다운로드, 수집 결과 병합

2-6-1. 수집 결과 다운로드

  • 각 머신에서 수집 결과 파일을 받는다.
  • 결과 파일 이름은 머신이름_seoulK_with_body.csv 이기 때문에, 여기에 맞춰서 코드가 작성되었다.

    2-6-2. 수집 결과 병합

  • 수집 결과 파일(csv)이 여러 개 쌓였을 텐데, 하나로 병합하자.
  • 결과 파일 하나하나가 몇백MB 단위이기 때문에 때문에 외부 툴을 사용하는 건 무리고, shell에서 바로 합치자.
  • awk ‘FNR > 1’ *.csv > merged_body.csv`
    • 모든 csv 파일들에서, 1행 이후의 내용을 가져와서, merged_body.csv로 만든다.
    • awk에 대한 상세 설명은 여기로
    • FNR : 현재 파일에서 현재 레코드의 순서 값.
      • ‘FNR > 1’ : 1번째 레코드보다 큰 것만. 따라서 header가 전부 빠진다.
      • 이 처리를 안 해주면 파일의 갯수만큼 header가 반복된다.
  • header는 따로 파일을 만들어서 보관하고 있다가, 따로 병합해 준다.

2-6-3. shell에서 csv 다루기 보너스

  • file.csv의 특정 열에서 중복 제거하기
    • awk -F "," '!x[$1]++' file.csv > file_unique.csv
    • -F : 필드 구분 문자 지정. csv이므로 ,를 지정함
    • '!x[$1]++'
      • $1 : 1번째 행.
      • !x[$1]++ : 어떻게 작동하는지는 여기를 참고하자. 간단히 요약하면, 처음 만나는 값에 대해서는 값이 x[value]가 0이고, 1이 증가한다. 0이 아닌 값은 출력하지 않는다.

주제 : 기업부도예측 with 머신러닝

1. 주제 선정 이유

  • 10년 전, 1년 간의 첫 주식 투자 결과 : 200만원 -> 200만원
  • 우량주를 사라고 하는데, 우량주의 기준이 도대체 무엇일까?
    • 펀더멘탈이 튼튼한 기업 -> 기업신용평가
  • 기업신용평가
    • 개인신용평가는 상대적으로 수치화, 모델화가 빠르게 이루어졌으나, 기업신용평가의 경우 최근까지 평가자의 주관적 의견이 중요한 요소였음 : 현재 뜨거운 주제 -> ‘기업’ 신용평가에 ML 탑재한 KB국민은행
    • 정형 데이터(재무, 주가)와 비정형 데이터(기사, 감사보고서)가 다수 공개되어 있는 주제

2. 부도의 기준

  • 한국금융연구원의 빅데이터를 이용한 딥러닝 기반의 기업 부도예측 연구 보고서의 기준 사용
    • 이 보고서에서는 아이디어와 접근방법을 참고하였음
    • 사용한 데이터에 대한 상세 설명 없음, 작성 코드 미공개, 학습모델 튜닝 과정 설명 없음, 모델 평가 수치 기재 미흡 등의 문제가 산재해 있어 좋은 보고서는 아님
  • 증권시장에서 ‘상장폐지’가 결정된 기업들 중 부도에 관련된 공시가 발생한 기업들을 부도 발생기업으로 인식하고 분석을 진행
    • ‘부도발생’, ‘화의절차개시신청’, ‘회사정리절차개시신청’, ‘감사인의 의견 거절’ 및 ‘은행거래정지’ 등 기업의 부실 및 지속 가능성이 심각하게 의심되는 사유로 발생한 상장폐지 사건을 부도로 정의
  • 상장폐지된 기업의 부도 여부는, 직접 사람의 손으로 정리하였다…

3. 프로젝트 목표

  • 3개월 후, 6개월 후, 1년 후에 어떤 기업이 부도를 맞이할 지 아닐 지 판단하는 모델 제작
    • 부도 확률을 수치로 나타낼 수 있다면 더욱 좋음
  • 웹으로 실제 서비스 수준의 시각화, 리포트 화면 제공
    • 원하는 기업을 검색하면, 해당 기업에 관련된 데이터를 시각화하여 리포트로 제공하는 서비스
